Build the LinkedIn OAuth integration: authorization redirect, callback handler with token exchange and encryption, LinkedIn API client with versioned headers, and a settings page where users can connect/disconnect their LinkedIn account and view connection status.

Purpose: Users need to connect their LinkedIn profile so the AI can publish content on their behalf. This plan handles the OAuth flow, secure token storage, and the UI for managing the connection.

NOTE: This plan only depends on Plan 01 (auth system). It does NOT depend on Plan 02 (onboarding/database). The LinkedIn OAuth callback stores tokens in the `platformAccounts` table from the Drizzle schema, but if Plan 02 has already run and pushed the schema, the table exists. If executing in parallel with Plan 02, the `npx drizzle-kit push` from Plan 02 Task 1 must complete first to create the tables. In practice, Plan 02 Task 1 (schema + DB push) will finish well before Plan 03 reaches the callback handler that writes to the database.

  <action>
    1. Create `src/lib/server/linkedin/tokens.ts` -- AES-256-GCM encryption:
       - Import `createCipheriv`, `createDecipheriv`, `randomBytes` from `crypto`
       - Import `TOKEN_ENCRYPTION_KEY` from `$env/static/private`
       - `const KEY = Buffer.from(TOKEN_ENCRYPTION_KEY, 'hex')` -- 32 bytes (256 bits)
       - `encrypt(text: string): string` -- generates random 16-byte IV, encrypts with aes-256-gcm, returns `iv:authTag:ciphertext` as hex-encoded string
       - `decrypt(encrypted: string): string` -- splits on ':', recreates decipher with IV and auth tag, returns plaintext
       - Follow the EXACT pattern from 02-RESEARCH.md Pattern 4

    2. Create `src/lib/server/linkedin/oauth.ts` -- OAuth flow helpers:
       - `LINKEDIN_SCOPES = 'openid profile email w_member_social'` (request all foreseeable scopes upfront per Pitfall 2)
       - `buildAuthorizationUrl(origin: string, state: string): string` -- constructs LinkedIn authorization URL with response_type=code, client_id, redirect_uri, scope, and state
       - `exchangeCodeForTokens(code: string, origin: string): Promise<LinkedInTokens>` -- POSTs to `https://www.linkedin.com/oauth/v2/accessToken` with grant_type=authorization_code, code, client_id, client_secret, redirect_uri. Returns `{ access_token, expires_in, refresh_token?, refresh_token_expires_in? }`
       - `fetchLinkedInProfile(accessToken: string): Promise<LinkedInProfile>` -- GET `https://api.linkedin.com/v2/userinfo` with Bearer token. Returns `{ sub, name, email, picture }`
       - Import LINKEDIN_CLIENT_ID and LINKEDIN_CLIENT_SECRET from `$env/static/private`

    3. Create `src/lib/server/linkedin/api.ts` -- LinkedIn API client:
       - `const LINKEDIN_API_VERSION = '202602'`
       - `linkedinHeaders(accessToken: string)`: returns headers object with Authorization Bearer, Content-Type application/json, Linkedin-Version, X-Restli-Protocol-Version 2.0.0
       - `createTextPost(accessToken: string, personId: string, text: string): Promise<string | null>` -- POST to /rest/posts, returns x-restli-id header
       - `initializeImageUpload(accessToken: string, ownerUrn: string)` -- POST to /rest/images?action=initializeUpload
       - `uploadImageBinary(uploadUrl: string, imageBuffer: Buffer, accessToken: string)` -- PUT binary to upload URL
       - `createImagePost(accessToken: string, authorUrn: string, text: string, imageUrn: string, altText?: string)` -- POST to /rest/posts with media content
       - `fetchOrganizations(accessToken: string): Promise<Organization[]>` -- GET /rest/organizationAcls to list company pages the user is admin of. Returns array of `{ id, name, avatarUrl }`. This is for future company page support.
       - Follow the EXACT patterns from 02-RESEARCH.md LinkedIn API Version Header Pattern and code examples

    4. Create `src/routes/api/oauth/linkedin/authorize/+server.ts`:
       - GET handler
       - Requires auth (locals.getUser(), 401 if null)
       - Reads `product_id` from URL search params
       - Generates cryptographic random state via `crypto.randomBytes(32).toString('hex')`
       - Stores state in httpOnly cookie `linkedin_oauth_state` (path: /, secure: true, sameSite: lax, maxAge: 600)
       - Stores product_id in httpOnly cookie `linkedin_oauth_product` (same settings)
       - Builds authorization URL via `buildAuthorizationUrl`
       - Redirects (302) to LinkedIn consent screen
       - Follow the EXACT pattern from 02-RESEARCH.md Pattern 3

    5. Create `src/routes/api/oauth/linkedin/callback/+server.ts`:
       - GET handler
       - Requires auth
       - Reads `code` and `state` from URL search params
       - Validates state against `linkedin_oauth_state` cookie -- return 400 if mismatch (CSRF protection)
       - Reads `product_id` from `linkedin_oauth_product` cookie
       - Exchanges code for tokens via `exchangeCodeForTokens`
       - Fetches user profile via `fetchLinkedInProfile`
       - Encrypts access_token and refresh_token via `encrypt()`
       - Checks if a platformAccount already exists for this product + platform + platformUserId. If so, update it (upsert pattern). If not, insert new record.
       - Uses Drizzle: `db.insert(platformAccounts).values({...}).onConflictDoNothing()` or check-then-insert pattern
       - Stores: productId, platform='linkedin', accountType='personal', platformUserId=profile.sub, platformUserName=profile.name, platformAvatarUrl=profile.picture, encrypted accessToken, encrypted refreshToken (if present), tokenExpiresAt (now + expires_in seconds), refreshTokenExpiresAt (if refresh_token_expires_in present), status='active'
       - Cleans up both OAuth cookies
       - Redirects (303) to `/dashboard/settings/linkedin?connected=true`
       - Follow the EXACT pattern from 02-RESEARCH.md LinkedIn OAuth callback example
  </action>
